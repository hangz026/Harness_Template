# Agent 行为公约模板 (AGENTS.md)

本文件定义 Agent 的通用工作方式，不写项目专属业务细节。

## 1. 文档职责

| 文档 | 职责 |
|---|---|
| `Rule.md` | 不可协商的项目硬约束 |
| `Spec.md` | 用户可见需求、输入输出、界面范围、交付物定义 |
| `Architecture.md` | 当前可执行的最小架构与演化边界 |
| `STATUS.md` | 当前项目状态与已批准方案的真实进展 |
| `CHANGELOG.md` | 重要变更历史 |
| `HARNESSLOG.md` | 仅模板仓库使用，记录 harness 骨架自身的演化 |
| `docs/` | 领域知识、公式、接口、外部依据 |
| `shared/` | 项目手动放入的共享资产区域，模板默认留空 |
| `lessons/` | 可复用的实现经验索引，详细约束见 `lessons/README.md` |

## 2. 冲突优先级

文档冲突时按以下顺序处理：

1. `Rule.md`
2. `Spec.md`
3. `Architecture.md`
4. `lessons/`
5. 其他文档

发现冲突必须先说明，再修正文档或方案，禁止自行忽略。

## 3. 工作原则

- 先审查，再写代码。
- 先确认边界，再提方案。
- 先复用已有内容，再考虑新建接口或新抽象。
- 不猜接口，不臆想业务，不越过规则红线。
- 不会就明确说明，不盲改，不糊涂重构。
- 文档必须各守其责，禁止越权承载其他文档职责。
- 若某项内容已在更合适的文档中定义，其他文档只做引用，不重复展开。
- 涉及结构、目录、分层和演化边界时，以 `Architecture.md` 为准。

## 4. 编码前审查

每次动手前，至少回答以下问题：

1. 这次是否真的需要写代码，还是只需澄清需求、修正文档、复用现有实现？
2. 现有代码、文档、库或共享资产里是否已经有可复用内容？
3. 本次是否涉及结构、目录、分层或演化边界；若涉及，是否已按 `Architecture.md` 审查？
4. 本次修改是否触碰 `Rule.md` 中的硬约束？
5. 当前边界是否足够清楚；若不清楚，是否应先提问再继续？
6. 若本次任务涉及特定技术主题，`harness/lessons/` 中是否已有相关经验约束？

如果以上问题没有明确答案，不进入编码。

## 5. 方案与审批

- 任何代码修改、结构调整、批量文档修改前，先给出简明方案。
- 方案至少包含：改什么、为什么、影响面、为什么这是当前最小方案。
- 用户明确同意后再修改。
- 修改完成后，等待用户审核，再进入下一步。
- 禁止跳过“方案 -> 同意 -> 修改 -> 审核”链路。
- 用户对某次特定文件、共享资产、红线约束或例外处理的批准，仅对当次明确范围有效。
- 禁止将单次批准推断为后续同文件、同类资产或同类例外的持续授权。
- 若后续再次触碰同一文件、同一共享资产或同一红线例外，必须重新单独申请并获得批准。

## 6. 代码、交付物与依赖约束

- 业务代码不得依赖 `harness/` 目录存在。
- 最终交付物默认应为可独立运行的业务项目本体，具体目录由 `Spec.md` 明确。
- `harness/` 默认属于协作、治理、知识与共享资产区域，不属于最终交付物，除非 `Spec.md` 明确要求随产品一起交付。
- 运行所需源码、资源、配置和第三方资产必须落入最终交付物目录，不能只留在 `harness/`。
- `shared/` 资产如需复制到业务目录，必须按规则处理。
- 常见能力优先使用成熟库，禁止重复造轮子。
- 除非经批准，不手写 CSV/JSON/XML 等常见格式解析器。

## 7. 任务拆分与验证

- 任务按最小可验证单元推进。
- 单次改动保持小而清晰，便于审查。
- 每次修改后立即验证当前目标，不把验证拖到最后。
- 优先验证核心逻辑，再做 UI 集成。

## 8. 沟通规则

- 先给结论，再给必要依据。
- 在不牺牲质量的前提下，输入输出尽量简洁。
- 边界不清时，先提最少量问题，不堆砌猜测。
- 发现规则冲突、需求缺口、实现风险时，立即留痕说明。

## 9. 文档更新规则

- 需求变更：更新 `Spec.md`
- 硬约束变更：更新 `Rule.md`
- 当前实现方式或演化边界变更：更新 `Architecture.md`
- 已批准任务完成或当前状态变化：更新 `STATUS.md`
- 重要变更：更新 `CHANGELOG.md`
- harness 骨架模板自身变化：仅在维护模板时更新 `HARNESSLOG.md`
- 新增可复用实现经验：更新 `lessons/`

项目实例化后默认不包含 `HARNESSLOG.md`。具体项目的 `harness/` 不应新建 `HARNESSLOG.md`，项目进展、项目变更和项目经验应分别进入 `STATUS.md`、`CHANGELOG.md` 或 `lessons/`。

## 10. 禁止行为

- 在 `AGENTS.md` 写入项目专属业务规则
- 跳过审批直接修改
- 一次性生成大量未经审查的代码
- 在多个文档重复写同一份算法、业务或架构细节
- 用猜测代替接口、需求或边界确认
- 在某份文档中承载本应属于另一份文档职责的内容

## 11. 文档维护说明

- `AGENTS.md` 应保持通用性，不随具体项目变化而大幅改写。
- 如发现项目特定约束，应写入 `Rule.md` 而非 `AGENTS.md`。
- `lessons/` 的详细定义、适用范围和使用方式以 `lessons/README.md` 为准。
- 本文件应保持精简，定期回顾是否仍然必要、准确、可复用。
